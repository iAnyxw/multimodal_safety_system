## 1. 项目背景

### 1.1 问题定义

互联网平台上存在大量用户生成内容（UGC），包括文本评论、图片、音频等。这些内容可能包含：

- **辱骂 / 人身攻击**（insult）
- **色情低俗**（obscene）
- **仇恨言论**（identity_hate）
- **威胁言论**（threat）
- **暴力图像 / 血腥场景**

传统审核方案通常是单模态的（只看文本 or 只看图片），无法处理多模态混合的复杂场景。本系统旨在构建一个**统一的多模态安全审核引擎**，同时支持文本、图像、音频三种模态的输入，并通过融合引擎输出一致的风险判定。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| **多模态覆盖** | 文本 + 图像 + 音频，三种模态统一入口 |

| **可解释** | 每个判定附带自然语言解释 |
| **可配置** | 阈值/策略/融合参数均可调，不硬编码 |
| **生产可用** | FastAPI 后端 + Streamlit 前端，支持 Docker 部署 |

## 3. 数据流

### 3.1 总体流程

```
用户输入 (text / image / audio)
        │
        ▼
┌─────────────────────┐
│ 🆕 Normalizer        │  【仅文本】对抗→干净中文：拼音/谐音还原 + 去空格 + emoji处理
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│   ModalityService    │  分发：text→XLM-R, image→CLIP, audio→Whisper+XLM-R
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│  各模态独立推理       │  并行执行，各自输出 (decision, score, labels)
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 🆕 Adversarial Guard │  【仅文本】残余对抗模式检测 → UNSAFE 降级为 REVIEW
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│   FusionEngine       │  多模态融合：一致boost / 冲突penalty
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│   PolicyEngine       │  分数→业务决策：>0.8 UNSAFE, >0.4 REVIEW, else SAFE
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│   Explainer          │  生成中文自然语言解释
└──────┬──────────────┘
       │
       ▼
    最终输出 {decision, score, explanation, details}
```

### 3.2 融合引擎逻辑

| 场景 | 条件 | 处理 | 示例 |
|------|------|------|------|
| 多模态一致危险 | ≥2 个模态 UNSAFE | score × 1.2 | 文本+图片都判不安全 |
| 多模态冲突 | 有 UNSAFE 也有 SAFE | score × 0.8 | 图片不安全但文本安全 |
| 单模态高置信 | 仅 1 模态，score > 0.7 | 原分上升 | 仅文本，score=0.85 |
| 单模态低置信 | 仅 1 模态，score ≤ 0.7 | 降级 REVIEW | 仅文本，score=0.3 |
| 全部安全 | 无 UNSAFE | score × 0.9 | 所有模态都判 SAFE |

### 3.3 策略引擎规则

| 优先级 | 分数范围 | 决策 | 标签 |
|--------|---------|------|------|
| 1 | ≥ 0.8 | UNSAFE | high_risk |
| 2 | ≥ 0.4 | REVIEW | medium_risk |
| 3 | < 0.4 | SAFE | low_risk |

## 4. 各模块详解

### 4.1 文本审核模块 (TextModule)

**模型**：XLM-RoBERTa-base

**标签**：3 类 — insult（辱骂）、obscene（淫秽）、identity_hate（身份仇恨）

**模型路径**：`checkpoints/xlmr_ch/`

**推理流程**：
```
text → tokenizer → model → sigmoid → threshold → (labels, score, decision)
```

- `score` = 所有标签中超过阈值的最大幅度：`max(prob - threshold, 0)`
- `UNSAFE_EXCEED = 0.15`：score > 0.15 才判 UNSAFE

### 4.2 图像审核模块 (ImageModel)

**双轨道设计**：

| 轨道 | 模型 | 职责 |
|------|------|------|
| 快速轨 | CLIP-ViT-Base-Patch32 | 全量图片初筛 |
| 深度轨 | Qwen2-VL-2B-Instruct | 疑图二次确认 |

**CLIP 多维度分类**（18 个文本提示词）：
- 色情（4）：pornographic, sexual, nude, anime/cartoon porn
- 暴力/威胁（9）：violent, fighting, blood, armed attack, shooting, threatening, horror/ghost, self-harm, animal abuse
- 安全（5）：normal daily, safe, fitness/fashion body, medical, gym workout

**三层决策体系**：

```
CLIP-Only:       0         0.3       0.6        1.0
                 ├──────────┼─────────┼──────────┤
                 │   SAFE   │ REVIEW  │ UNSAFE   │

Dual-Track:      0         0.3       0.6        1.0
(触发判定)        ├──────────┼─────────┼──────────┤
                 │ 直接 SAFE │ 送 Qwen  │ 直接UNSAFE│

Qwen 内部:       0         0.4       0.6        1.0
                 ├──────────┼─────────┼──────────┤
                 │   SAFE   │ REVIEW  │ UNSAFE   │
```

**设计原则**：
- CLIP 0.3-0.6 判 REVIEW，表示「拿不准」，对应双轨中送 Qwen 的区间
- Qwen 0.4-0.6 判 REVIEW，比 CLIP 窄——大模型判断更准，放行空间更大
- CLIP >0.5 但 Qwen <0.4 时，采用保守策略保留 REVIEW，不降为 SAFE

### 4.3 音频审核模块 (AudioModule)

**流水线**：
```
音频文件 → 降噪增强 → Whisper 转写 → XLM-R 文本审核 → 风险判定
```

**预处理**：
- `noisereduce` 降噪
- `librosa` 归一化
- 音频质量检查（能量/SNR/静音比）

**模型**：Whisper-base（语音识别）+ XLM-RoBERTa（文本审核，复用文本模块）

### 4.4 VLM 深度审核 (VLMModule)

**模型**：Qwen2-VL-2B-Instruct

**触发条件**：CLIP 分数在 0.4~0.6 之间的疑图

**审核方式**：通过精心设计的 system prompt，让 VLM 输出结构化 JSON，包含 porn / violence / safe 三个维度的风险评分。

**特点**：能理解图片的真实语境（如区分"健身"和"裸露"、"泳装"和"色情"）。

#### 6.1.1 对抗样本鲁棒性测试 ✅

**测试集**：52 条手工构造对抗样本，覆盖 5 种攻击类型。脚本：`scripts/test_adversarial.py`

**结果总览**：

| 指标 | 数值 |
|------|------|
| Unsafe 样本 | 37 条 → 全部正确拦截 |
| Unsafe 召回率 | **100.0%**（0 漏放） |
| Safe 样本 | 15 条 → 仅 1 条正确放行 |
| Safe 准确率 | **6.7%**（14 条误拦） |
| 总体正确率 | 38/52 = **73.1%** |

**按对抗类型统计**：

| 类型 | 样本数 | 正确 | 准确率 | 误拦 | 漏放 |
|------|--------|------|--------|------|------|
| 1. 拼音替代 | 12 | 10 | 83.3% | 2 | 0 |
| 2. 空格分隔 | 10 | 7 | 70.0% | 3 | 0 |
| 3. 谐音替换 | 12 | 10 | 83.3% | 2 | 0 |
| 4. emoji 干扰 | 8 | 6 | 75.0% | 2 | 0 |
| 5. 语境暧昧 | 10 | 5 | **50.0%** | 5 | 0 |
| **总计** | **52** | **38** | **73.1%** | **14** | **0** |

**对比：干净文本 vs 对抗样本**：

| 场景 | Safe 准确率 | Unsafe 召回率 |
|------|------------|---------------|
| 干净文本（验证集） | **100.0%** | — |
| 对抗样本 | **6.7%** | **100.0%** |

**根因分析**：
- ⚠️ 模型极度激进——所有被误拦的 safe 对抗样本 insult 概率均 > 0.92
- 唯一幸存的 safe 样本 `今天😄很开心`（emoji 破坏了字符关联模式）
- 核心问题：训练集中从未出现 safe 标签的拼音/空格/谐音变体，模型对未知 pattern 默认偏向 unsafe

> ⚠️ 初步结论：模型在干净文本上无误伤（safe F1=1.00），但对对抗变体误拦率极高。**14 条误拦全集中在 Safe→Unsafe 方向，0 条漏放**——说明模型能检测异常 pattern 但缺乏精确校准。

#### 6.1.2 对抗增强训练尝试（消融） ⚠️ 失败

**方案**：生成 ~600 条对抗增强样本（safe 对抗变体 + unsafe 对抗变体），混入训练集重训 XLM-R。

**结果**：

| 指标 | 原始模型 | 增强后 v1 | 增强后 v2（去 safe 拼音变换） |
|------|---------|-----------|------------------------------|
| Safe 对抗准确率 | **6.7%** | 46.7% | 60.0% |
| Unsafe 对抗召回率 | **100%** | 56.8% ❌ | 51.4% ❌ |
| 干净文本 Micro F1 | 0.954 | — | 0.946 |

> ❌ **结论：数据增强无法解决对抗鲁棒性问题。** Safe 准确率每提升一点，Unsafe 召回率就崩塌。根因是 XLM-R 的 subword tokenizer 对拼音/空格/谐音产生的是无意义 token，模型无法从垃圾 token 中学到有意义的 safe/unsafe 边界。**这是一个 tokenizer 层面的信息瓶颈，而非数据量不足。**

#### 6.1.3 Adversarial Guard（系统级修复） ✅

**方案**：不改模型，在推理层增加对抗模式检测（`scripts/adversarial_guard.py`）。检测到对抗模式 + 模型判 UNSAFE → 降级为 REVIEW。

**Guard 检测规则**：

| 规则 | 检测内容 | 示例 |
|------|---------|------|
| 纯拼音 | 拼音字符占比 > 50% | `nmsl`, `cao ni ma` |
| 空格分隔 | 中文字间空格密度异常 | `你 去 死`, `你 好 吗` |
| 谐音检测 | 含常见对抗谐音词 | `煞笔`, `草泥马`, `碧池` |
| emoji 干扰 | 中文 + emoji + 短文本 | `🔪你`, `我😊你` |
| 短纯拼音 | 2-10 个纯字母 | `sb`, `cnm`, `nmd` |

**Guard 效果**：

| 指标 | 原始模型 | 原始模型 + Guard |
|------|---------|-----------------|
| Safe 对抗样本误拦数 | 14/15 直接 UNSAFE | **7/15 降为 REVIEW，7/15 仍 UNSAFE** |
| Unsafe 召回率 | **100.0%** | **100.0%**（不变） |
| 等效 Safe 准确率 | 6.7% | **53.3%** |

> ✅ **结论：Guard 在不牺牲 Unsafe 召回率（仍 100%）的前提下，将 50% 的误拦从「直接拦截」降级为「人工复审」，大幅减少误杀。** 这是一个安全保守的方案——宁可 REVIEW 也不放过任何不安全内容。脚本：`scripts/adversarial_guard.py`。

#### 6.1.4 预处理归一化（Pre-Normalization） ✅ 推荐

**核心思路**：模型在干净中文上 Safe F1=1.00。与其让模型面对垃圾 token（拼音/空格/谐音/emoji），不如在推理前先归一化为干净中文。

**归一化流水线**（`scripts/normalizer.py`）：

```
对抗输入  →  谐音替换  →  拼音转中文  →  去空格  →  emoji替换/删除  →  干净中文  →  XLM-R 推理
```

**归一化规则**：

| 规则 | 输入 | 输出 |
|------|------|------|
| 拼音→中文（已知映射） | `nmsl`, `nihaoma` | `你妈死了`, `你好吗` |
| 谐音→原字 | `煞笔`, `草泥马` | `傻逼`, `操你妈` |
| 去中文字间空格 | `你 去 死`, `你 好 吗` | `你去死`, `你好吗` |
| 危险 emoji→文字 | `🔪你`, `去💀吧` | `杀你`, `去死吧` |
| 安全 emoji→删除 | `我😊你`, `😍爱你` | `我你`, `爱你` |

**三方案对比**：

| 方案 | Safe 准确率 | Unsafe 召回率 | 总体 | 误拦 | 漏放 |
|------|-----------|-------------|------|------|------|
| 原始模型 | 6.7% | **100%** | 73.1% | 14 | 0 |
| + Guard | 53.3%* | **100%** | — | 7→REVIEW | 0 |
| 🧹 + 归一化 | 13.3% | **100%** | 75.0% | 13 | 0 |

> \* Guard: 误拦转 REVIEW 而非放行。归一化 Safe 准确率提升有限（6.7%→13.3%），因为 10/15 的 safe 对抗样本属于「语境暧昧」类（如`杀了我吧`、`我想死`），本身是干净中文，语义层面难以区分。

**最终推荐：归一化 + Guard 组合**：

```
输入 → 🧹 归一化 → XLM-R 推理 → 🛡️ Guard → Policy → 输出
```

| 层 | 解决什么问题 |
|----|------------|
| 🧹 归一化 | 拼音/空格/谐音/emoji 等**正交对抗** → 还原为干净中文 |
| 🛡️ Guard | 归一化后仍被误拦的残余对抗文本 → 降级 REVIEW |

> ✅ **归一化 + Guard 组合在保持 Unsafe 召回率 100% 的前提下，最大化减少对 safe 对抗文本的误杀。** 脚本：`scripts/normalizer.py`（归一化）+ `scripts/adversarial_guard.py`（Guard）。

### 6.2 图像模型 ✅

| 组件 | 模型 | 说明 |
|------|------|------|
| 快速轨 | CLIP-ViT-Base-Patch32 | 18 维度多标签分类 |
| 深度轨 | Qwen2-VL-2B-Instruct | VLM 结构化审核 |

**测试集**：206 张标注图片（Safe: 100 / Unsafe: 106）

#### 6.2.1 CLIP-Only（快速轨单独）

| 指标 | 数值 |
|------|------|
| 完全正确 | 107/206 = **51.9%**（Safe 57 + Unsafe 50） |
| 复审兜底（送 Qwen） | 90/206 = **43.7%**（Safe 36 + Unsafe 54） |
| 真正误判 | 9/206 = **4.4%** |
| — 误拦（Safe→Unsafe） | 7 张 |
| — 漏放（Unsafe→Safe） | 2 张 |
| 总耗时 | **7.3s** |

> CLIP 单独使用时，仅 4.4% 真正误判；43.7% 落入 REVIEW 区间需送 Qwen 二次确认。

#### 6.2.2 Qwen-Only（深度轨单独）

| 指标 | 数值 |
|------|------|
| Safe 准确率 | 95/100 = **95.0%** |
| Unsafe 召回率 | 97/106 = **91.5%** |
| 总体正确率 | 192/206 = **93.2%** |
| 总耗时 | **62.4s** |

> Qwen 单独审核全部 206 张可达 93.2% 正确率，但耗时 62.4s（平均 0.30s/张），不适合全量实时。

#### 6.2.3 CLIP+Qwen（双轨联合）

| 指标 | 数值 |
|------|------|
| CLIP-Only Safe 正确率 | 91/100 = **91.0%** |
| CLIP+Qwen Unsafe 召回率 | 98/106 = **92.5%** |
| 总体正确率 | 189/206 = **91.7%** |
| Qwen 触发次数 | 90/206 = **43.7%** |
| 总耗时 | **39.4s** |

> 双轨模式下 Qwen 仅处理 43.7% 的疑图，总耗时 39.4s（相比 Qwen-Only 的 62.4s 减少 37%），总体正确率 91.7%，接近 Qwen-Only 的 93.2%。

#### 6.2.4 消融实验（Ablation Study）

| 实验配置 | CLIP | Qwen | 正确率 | 耗时 | Δ正确率 | Δ耗时 |
|----------|------|------|--------|------|---------|------|
| **CLIP+Qwen（完整系统）** | ✅ | ✅ | **91.7%** | **39.4s** | — | — |
| Qwen-Only（消融 CLIP） | ❌ | ✅ | 93.2% | 62.4s | +1.5% | **+58.4%** ⚠️ |
| CLIP-Only（消融 Qwen） | ✅ | ❌ | 51.9%* | 7.3s | **−39.8%** ⚠️ | −81.5% |

> \* CLIP-Only 另有 43.7% 落入 REVIEW 区间无法自动决策，实际可自动决策的正确率为 107/(206−90)=92.2%

**消融结论**：

| 移除组件 | 主要影响 | 量化 |
|----------|----------|------|
| 移除 **CLIP**（只用 Qwen） | 速度大幅下降 | 耗时 +58.4%（39.4s → 62.4s） |
| 移除 **Qwen**（只用 CLIP） | 精度大幅下降 | 正确率 −39.8%（91.7% → 51.9%），且 43.7% 需人工介入 |

> ✅ **CLIP 贡献速度**（去掉后慢 58%），**Qwen 贡献精度**（去掉后 43.7% 无法自动决策）。双轨联合实现了精度与速度的最优折中。

cd multimodal_safety_system

### 8.2 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|

| POST | `/predict/audio` | 音频审核（文件上传） |
| POST | `/predict/multimodal` | 多模态融合审核 |

### 8.3 配置管理

所有配置集中在 `backend/config/settings.py`，支持 `.env` 文件覆盖：

```python
# 关键可调参数
TEXT_THRESHOLDS = {"insult": 0.5, "obscene": 0.5, ...}  # 各标签阈值
TEXT_UNSAFE_EXCEED = 0.15                                 # UNSAFE 判定门槛
FUSION_AGREEMENT_BOOST = 1.2                              # 多模态一致强化系数
FUSION_CONFLICT_PENALTY = 0.8                             # 冲突降权系数
POLICY_RULES = [...]                                      # 策略规则
```
